## Phase 1: Foundation

**Requirements:** INFRA-01, INFRA-02, INFRA-03, INFRA-04
**Status:** Human Verification Required — Plans 01 and 02 complete (INFRA-01, INFRA-02 verified ✓); INFRA-03/04 need db:migrate + npm run dev smoke test (see `01-HUMAN-UAT.md`)

### Rationale

Every line of feature code written in Phase 2 and 3 depends on a working Drizzle schema, a confirmed Postgres connection, and correctly installed packages. Import path bugs (the most common AI SDK failure mode) and connection pool bugs are cheapest to catch on day zero and most expensive to discover mid-stream. Establishing migrations with `drizzle-kit generate` + `migrate` from the first day ensures no one on the team ever runs `drizzle-kit push` and loses migration history.

### What Gets Built

- Next.js App Router project with TypeScript and all packages installed at pinned versions
- `lib/db/index.ts` — Drizzle client with `globalThis` singleton guard (prevents connection exhaustion on hot reload)
- `lib/db/schema.ts` — `chats` and `messages` table definitions with foreign key constraint and index on `messages.chat_id`
- `drizzle/` — migration files generated by `drizzle-kit generate` and applied by `drizzle-kit migrate`; committed to the repo
- `.env.local.example` — template with `OPENAI_API_KEY` and `DATABASE_URL` slots, no hardcoded values
- `app/api/chat/route.ts` stub — correct imports locked in (`streamText` from `"ai"`, `useChat` from `"ai/react"`, provider from `"@ai-sdk/openai"`), `export const dynamic = "force-dynamic"` set

### Success Criteria

1. `npm run dev` starts without error after cloning the repo and exporting the two required environment variables — no missing dependency or missing env crashes.
2. Running `drizzle-kit migrate` against a fresh Postgres instance creates the `chats` and `messages` tables with the correct columns, foreign key, and index — verifiable by inspecting the DB schema.
3. A Server Component can execute a query against Postgres (e.g., list all chats) and return without error — confirming the Drizzle client singleton is wired correctly.
4. No environment value is hardcoded in any source file — `OPENAI_API_KEY` and `DATABASE_URL` are read exclusively from environment variables at runtime.

### Requirements Covered

| ID | Description |
|----|-------------|
| INFRA-01 | App reads `OPENAI_API_KEY` and `DATABASE_URL` from environment variables |
| INFRA-02 | Drizzle ORM schema defined with `chats` and `messages` tables |
| INFRA-03 | DB migrations managed with `drizzle-kit generate` + `migrate`, not push |
| INFRA-04 | App runs locally with `npm run dev` after setting environment variables |

## Phase 2: Core Loop

**Requirements:** MSG-01, MSG-02, MSG-03, MSG-04, MSG-05, PERS-01, PERS-02, PERS-03, PERS-04, RELY-01, RELY-02, RELY-03
**Status:** In Progress — Plans 02-01 (db-query-layer) and 02-02 (route-handler) complete

### Rationale

Streaming, persistence, and retry are not independently shippable units — a streamed response that is not persisted is incomplete, and persistence without a working stream is untestable. They are developed together as the core request/response loop. This is the highest-risk phase: five distinct persistence failure modes live here (save-before-stream causing hang, user message lost on LLM failure, silent `onFinish` DB error, message duplication across retries, response buffered instead of streamed). All five are preventable but only if addressed deliberately. Verifying `Transfer-Encoding: chunked` in DevTools before building any UI on top of the stream is a hard gate.

### What Gets Built

- `app/api/chat/route.ts` — Route Handler calling `streamText`, returning `result.toDataStreamResponse()`; saves the user message to DB as the first operation before calling the LLM; saves the assistant response inside `onFinish` with try/catch and explicit error logging
- `lib/db/queries.ts` — typed CRUD functions: `createChat`, `getChats`, `getChat`, `getMessages`, `createMessage`, `updateChat`, `deleteChat`
- Retry wrapper around the LLM call — minimum 2 retries with exponential backoff on transient failures (timeout, 5xx, network error); user message is saved once outside the retry loop
- `components/ChatInterface.tsx` — `"use client"` component owning `useChat`; accepts `chatId` and `initialMessages` props; sends `chatId` in `useChat`'s `body` option so the Route Handler can associate messages with the correct conversation
- `components/MessageList.tsx` — renders messages from `useChat` in streaming-aware fashion
- `components/MessageInput.tsx` — textarea + submit; disabled while `isLoading`; renders `error` state inline when set
- `app/chat/[chatId]/page.tsx` (working stub) — Server Component that fetches messages from Postgres and passes them as `initialMessages` to `ChatInterface`

### Success Criteria

1. A user types a message and sees the assistant's response appear token-by-token in the browser — the first token arrives before the full response is complete, confirmed in the DevTools Network tab as `Transfer-Encoding: chunked` with incremental data frames.
2. After a conversation ends, a full page reload restores every user and assistant message from Postgres — no messages are lost, duplicated, or out of order.
3. A loading indicator (spinner or disabled input) is visible throughout the assistant's response and disappears exactly when the response is complete.
4. Simulating a transient LLM failure (e.g., network timeout on the first attempt) triggers automatic retries; the user message appears exactly once in the database regardless of how many retry attempts were made.
5. If the LLM call fails after all retries are exhausted, an error message is displayed in the chat UI — not a blank screen, not a silent failure, not an unhandled exception.

### Requirements Covered

| ID | Description |
|----|-------------|
| MSG-01 | User can type a message and send it to the LLM |
| MSG-02 | Assistant response renders in streaming mode — tokens appear as they arrive, not after full completion |
| MSG-03 | Multi-turn conversation supported — full message history sent to the LLM on each turn |
| MSG-04 | User sees a loading indicator while the assistant is responding |
| MSG-05 | User sees an error message if the LLM call fails |
| PERS-01 | Chat records are stored in Postgres (create chat on new conversation) |
| PERS-02 | Each user message is persisted to Postgres before the LLM call |
| PERS-03 | Each assistant response is persisted to Postgres after streaming completes via `onFinish` |
| PERS-04 | Fetching a conversation's messages from Postgres and loading them into the chat view works correctly |
| RELY-01 | Backend retries LLM calls on transient failure — at minimum 2 retries with backoff |
| RELY-02 | Retry logic does not duplicate messages — user message saved once before retry attempts begin |
| RELY-03 | Errors after all retries are exhausted are surfaced to the user, not swallowed |

## Phase 3: Conversations UI

**Requirements:** CONV-01, CONV-02, CONV-03, CONV-04, CONV-05
**Status:** Not started

### Rationale

With a proven streaming and persistence core, the UI shell can be built on a stable foundation. The dynamic route `/chat/[chatId]` is a hard dependency for every sidebar link — routing must exist before navigation is functional. All DB queries needed by the sidebar (list chats, delete chat, update title) were already written in Phase 2; this phase wires them into components and layout. The stream-leak-on-navigation bug (tokens from chat A appearing in chat B) must be addressed here as a first-class concern, not an afterthought.

### What Gets Built

- `app/layout.tsx` — root layout with sidebar slot and main content slot
- `app/page.tsx` — root entry point: creates a new chat and redirects to `/chat/<id>`, so the user always lands in a conversation
- `app/chat/[chatId]/page.tsx` (completed) — Server Component fetching the chat record and its messages; passes `initialMessages` to `ChatInterface` and validates the `chatId` param exists
- `components/Sidebar.tsx` — Server Component that fetches all chats; renders a scrollable list with active-chat highlight; links each entry to `/chat/[chatId]`
- New chat action — Server Action or form that inserts a chat row and navigates to the new chat; sidebar updates to reflect it
- Rename conversation — inline edit or form that calls `updateChat` and persists the new title in Postgres
- Delete conversation — calls `deleteChat` (cascade removes all child messages); sidebar refreshes to remove the entry
- `useEffect` cleanup in `ChatInterface` — calls `useChat.stop()` when `chatId` prop changes or the component unmounts, preventing stream tokens from a previous chat leaking into the current one

### Success Criteria

1. A sidebar is visible at all times listing all past conversations by title; clicking any entry navigates to that conversation and displays its complete message history.
2. Clicking "New Chat" creates a fresh conversation, navigates to it, and the sidebar immediately shows the new entry without requiring a manual refresh.
3. Renaming a conversation updates the title in the sidebar; the new name is still present after a full page reload, confirming it was persisted to Postgres.
4. Deleting a conversation removes it from the sidebar and removes all of its messages from the database — a direct DB query after deletion confirms no orphaned message rows remain.
5. Switching to a different conversation while an assistant response is mid-stream does not cause tokens from the previous conversation to appear in the newly selected conversation.

### Requirements Covered

| ID | Description |
|----|-------------|
| CONV-01 | User can create a new chat (new conversation session) |
| CONV-02 | User can view a list of all past conversations in a sidebar |
| CONV-03 | User can open an existing conversation and see its full message history |
| CONV-04 | User can update (rename) a conversation |
| CONV-05 | User can delete a conversation and all its messages |

## Phase Sequencing Rationale

**Phase 1 before Phase 2:** The Drizzle schema, DB client, and migration workflow are hard dependencies for every DB operation in Phase 2. Schema changes after persistence code is written cause migration drift. Getting import paths correct at project initialization (before any feature code exists) is also the cheapest time to catch the most common AI SDK failure mode.

**Phase 2 before Phase 3:** The sidebar renders links to `/chat/[chatId]` — the dynamic route segment and all DB queries backing it are built in Phase 2. Building routing before the persistence layer exists creates throwaway stubs. More importantly, the five persistence failure modes in Phase 2 are easiest to debug before any UI complexity is layered on top.

**MSG, PERS, and RELY bundled in Phase 2:** These three groups are not independently shippable. Streaming without persistence is a demo that loses all data. Persistence without streaming is untestable (no response arrives). Reliability (retry) is only meaningful in the context of a working send/receive/persist loop. They are one cohesive unit of work.
