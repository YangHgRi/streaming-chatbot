## 1. Spacing / Layout

### 1.1 Tailwind Spacing Scale in Use

All values are multiples of 4px. Tokens below are the full set observed in the codebase; Phase 3 does not introduce new tokens — it reuses the existing scale.

| Token | px value | Tailwind class | Phase 3 usage |
|-------|----------|----------------|---------------|
| xs | 4px | `p-1`, `gap-1` | Loading dot gaps (`gap-1`); tight icon padding |
| sm | 8px | `p-2`, `gap-2`, `px-2 py-2` | Sidebar nav outer padding (`p-2`); message input gap (`gap-2`); sidebar list item padding (`px-2 py-2`) |
| sm+ | 12px | `p-3`, `px-3 py-2`, `px-3 py-3` | Textarea inner padding (`p-3`); sidebar chat-item link padding (`px-3 py-2`) |
| md | 16px | `p-4`, `px-4 py-2` | Message list padding; input form padding; message bubble padding (`px-4 py-2`) |
| lg | 24px | `px-6 py-4`, `px-6` | Chat page header (`px-6 py-4`); home page body padding |
| xl | 32px | `px-8 py-3`, `space-y-6` | Home page primary CTA; vertical section separation |

Exceptions: `max-w-[80%]` on message bubbles — arbitrary value, not a spacing token.

### 1.2 Root Layout Structure

Replace current `<body className="min-h-full flex flex-col">` with a two-column horizontal flex layout:

```
<html class="h-full antialiased">               ← already correct, preserve font variables
  <body class="h-full flex overflow-hidden">    ← CHANGED from "min-h-full flex flex-col"
    <aside>  ← Sidebar — w-64, fixed width       }
    </aside>                                     } side by side
    <main class="flex-1 overflow-hidden">       }
      {children}                                ← page fills this slot
    </main>
  </body>
</html>
```

**Critical classes:**
- `h-full` on `<html>` and `<body>` — already set on `<html>`; must be on `<body>` too for full-height sidebar
- `overflow-hidden` on `<body>` — prevents double scrollbars; each column manages its own scroll
- `flex-1 overflow-hidden` on `<main>` — fills remaining width; inner content scrolls independently
- `flex-shrink-0` on `<aside>` — sidebar never compresses below `w-64`

### 1.3 Sidebar Internal Layout

```
<aside class="w-64 flex-shrink-0 bg-gray-900 text-white flex flex-col h-full">
  ┌─────────────────────────────────┐  p-4 border-b border-gray-700
  │  [Plus icon]  New Chat           │  ← header zone / action button
  └─────────────────────────────────┘
  ┌─────────────────────────────────┐  flex-1 overflow-y-auto p-2 space-y-1
  │  Chat item (active)              │  ← bg-gray-700
  │  Chat item                       │  ← hover:bg-gray-800
  │  Chat item                       │
  │  ...                             │
  └─────────────────────────────────┘
</aside>
```

Sidebar width: `w-64` (256px / 16rem) — conventional ChatGPT-style width.

### 1.4 Chat Page Layout Adjustment

The current `app/chat/[chatId]/page.tsx` uses `h-screen` which conflicts with the root layout's flex container. Replace:

```
BEFORE:  <main class="flex flex-col h-screen bg-gray-50">
AFTER:   <div class="flex flex-col h-full bg-gray-50">
```

The page becomes the `{children}` slot inside `<main class="flex-1 overflow-hidden">` — it must use `h-full` to fill its container, not `h-screen`.

Inner structure of the chat page (Phase 3 target):

```
<div class="flex flex-col h-full bg-gray-50">
  <header class="border-b border-gray-200 bg-white px-6 py-4 flex-shrink-0">
    <h1 class="text-lg font-semibold text-gray-900">{chat.title}</h1>  ← dynamic title
  </header>
  <div class="flex-1 overflow-hidden">
    <ChatInterface chatId={chatId} initialMessages={initialMessages} />
  </div>
</div>
```

`flex-shrink-0` on `<header>` prevents the header from being squeezed when the message list fills.

### 1.5 Sidebar Chat List Item Layout

Each chat entry is a horizontal flex row:

```
<div class="group flex items-center gap-1 rounded-lg [active|hover classes]">
  <Link class="flex-1 min-w-0 px-3 py-2 text-sm truncate">  ← title link
    {chat.title}
  </Link>
  <div class="hidden group-hover:flex items-center gap-1 pr-2 flex-shrink-0">
    [Pencil icon button]  ← rename trigger
    [Trash2 icon button]  ← delete trigger (or confirm/cancel when isPending)
  </div>
</div>
```

- `flex-1 min-w-0` on the link is required for `truncate` to work inside a flex parent (`min-w-0` overrides flexbox's default min-width: auto).
- Action icons are revealed via `group-hover:flex` — hidden at rest, visible on row hover.
- When rename edit mode is active OR delete confirmation is pending, action icons stay visible (not hidden).

### 2.3 Sidebar Typography Specifics

All sidebar text uses Geist Sans. Color contrast on `bg-gray-900`:

| Element | Classes | Contrast role |
|---------|---------|---------------|
| Active chat title | `text-white` | Primary — full contrast |
| Inactive chat title | `text-gray-100` | Near-primary — readable at rest |
| Muted / hovered title | `text-gray-300` | Secondary — still WCAG AA |
| New Chat button label | `text-white text-sm font-medium` | Full contrast on bg-gray-800 |
| Confirm/Cancel labels | `text-xs` — red-400 / gray-400 | Destructive / neutral affordance |

### 2.4 Rename Input Typography

When a chat title enters edit mode, the `<input>` must visually match the title text it replaces:

```
<input class="text-sm text-white bg-transparent border border-gray-500 rounded
               px-1 focus:outline-none focus:border-blue-400 w-full min-w-0" />
```

Same `text-sm` size as the title. `bg-transparent` keeps the sidebar background visible through the field. `focus:border-blue-400` — accent color on the dark background (blue-400 is lighter than blue-600 and visible on gray-900).

### 4.4 Rename Flow — Click-to-Edit Inline

Confirmed design decision Q2: **click-to-edit inline — title becomes `<input>` on click, Enter/blur submits.**

**Interaction sequence:**
1. User clicks Pencil icon button → `isEditing` state set to `true`
2. Title `<Link>` replaced by `<form>` containing `<input>` pre-filled with `chat.title`
3. User edits → presses Enter or blurs → form submits `renameChatAction`
4. User presses Escape → `isEditing` set back to `false` (no submit)
5. On submit → `isEditing` false; sidebar re-fetches via `revalidatePath`

**Inline form markup (edit mode):**

```
<div class="group flex items-center gap-1 rounded-lg bg-gray-700">  ← highlight while editing
  <form action={boundRenameChatAction} class="flex-1 min-w-0 px-2">
    <input
      type="text"
      name="title"
      defaultValue={chat.title}
      autoFocus
      class="w-full text-sm text-white bg-transparent border border-gray-500
             rounded px-1 py-1 focus:outline-none focus:border-blue-400 min-w-0"
      onKeyDown={(e) => e.key === 'Escape' && setIsEditing(false)}
    />
  </form>
  <div class="flex items-center gap-1 pr-2 flex-shrink-0">
    <button type="button" onClick={() => setIsEditing(false)}
            class="p-1 rounded text-gray-400 hover:text-gray-200">
      ✕  {/* or X icon */}
    </button>
  </div>
</div>
```

- `autoFocus` on the input ensures keyboard is ready immediately.
- `defaultValue` (not `value`) keeps it uncontrolled — native form submission reads the value.
- Form is submitted on blur via `onBlur` on the input triggering `form.requestSubmit()`.
- Escape key sets `isEditing(false)` without submitting.
- `renameChatAction` is bound with `chatId`: `renameChatAction.bind(null, chat.id)`.

**State managed in `SidebarClient` (or a per-item sub-component):**

```ts
const [editingId, setEditingId] = useState<string | null>(null);
// editingId === chat.id → show input for that item
```

### 4.5 Delete Flow — Inline Confirm / Cancel Toggle

Confirmed design decision Q3: **inline Confirm / Cancel toggle — most polished, requires client state.**

**Interaction sequence:**
1. User clicks Trash2 icon button → `confirmingId` state set to `chat.id`
2. Row action area replaces icons with two text buttons: "Delete" (red) and "Cancel"
3. Click "Delete" → calls `deleteChatAction(chat.id)` → sidebar re-fetches; redirect to `/`
4. Click "Cancel" → `confirmingId` set back to `null`
5. Clicking away from the row → does NOT auto-dismiss (explicit Cancel required)

**Confirm / Cancel markup (pending state):**

```
<div class="flex items-center gap-2 pr-2 flex-shrink-0">
  <button
    onClick={() => startTransition(() => deleteChatAction(chat.id))}
    class="text-xs text-red-400 hover:text-red-300 font-medium">
    Delete
  </button>
  <button
    onClick={() => setConfirmingId(null)}
    class="text-xs text-gray-400 hover:text-gray-300">
    Cancel
  </button>
</div>
```

- `startTransition` wraps the delete call to expose `isPending` for optional loading state.
- While `isPending`: disable both buttons (`disabled:opacity-50 disabled:pointer-events-none`).
- `deleteChatAction` is called directly (not via `<form>`) because the user already confirmed.
- After `deleteChatAction` calls `redirect('/')`, the sidebar will reflect the deletion on next render.

**State managed in `SidebarClient`:**

```ts
const [confirmingId, setConfirmingId] = useState<string | null>(null);
const [isPending, startTransition] = useTransition();
```

### 4.6 `ChatInterface` — Stream Cleanup

Add `useEffect` with `stop` to prevent token leak when switching conversations mid-stream (Phase 3 gate criterion):

```ts
const { messages, sendMessage, status, error, stop } = useChat({
  id: chatId,
  messages: initialMessages,
});

// Abort any in-progress stream when chatId changes or component unmounts.
// Prevents tokens from conversation A leaking into conversation B.
useEffect(() => {
  return () => {
    stop();  // fire-and-forget: stop() returns Promise<void>; cleanup must be sync
  };
}, [chatId, stop]);
```

- `stop` is included in deps — it is recreated when `chatId` changes (AI SDK v6 creates a new Chat instance per `id`), so this correctly re-registers cleanup for each chat.
- React calls the cleanup **before** running the effect with the new `chatId` — stop fires first, then the new chat initializes. Order is guaranteed.
- Do NOT `await stop()` in the cleanup — effect cleanup functions must be synchronous.

### 4.7 `app/page.tsx` — Auto-Redirect

Replace current button-click pattern with auto-redirect on load:

```tsx
import { redirect } from 'next/navigation';
import { createChat } from '@/lib/db/queries';

export default async function HomePage() {
  const chat = await createChat();
  redirect(`/chat/${chat.id}`);
}
```

- `redirect()` must NOT be inside a try/catch block — it throws a special Next.js error internally.
- Every visit to `/` creates a new chat and redirects. This is acceptable for a single-user demo (orphan-chat risk is cosmetic, not functional).
- The sidebar's "New Chat" button is the primary interaction for subsequent chats; the root page is only a fallback entry point.

### 4.8 `app/actions.ts` — Server Actions Module

New file at `src/app/actions.ts`. All three mutations live here:

```ts
'use server';
import { redirect } from 'next/navigation';
import { revalidatePath } from 'next/cache';
import { createChat, updateChat, deleteChat } from '@/lib/db/queries';

export async function createChatAction() {
  const chat = await createChat();
  revalidatePath('/', 'layout');   // bust Router Cache for layout subtree (sidebar re-fetches)
  redirect(`/chat/${chat.id}`);    // outside try/catch
}

export async function renameChatAction(chatId: string, formData: FormData) {
  const title = formData.get('title') as string;
  if (!title?.trim()) return;
  await updateChat(chatId, { title: title.trim() });
  revalidatePath('/', 'layout');
  // No redirect — user stays on current chat; sidebar title updates
}

export async function deleteChatAction(chatId: string) {
  await deleteChat(chatId);        // CASCADE removes all child messages automatically
  revalidatePath('/', 'layout');
  redirect('/');                   // navigate away from deleted chat
}
```

`revalidatePath('/', 'layout')` invalidates the Router Cache for the root layout subtree, causing `Sidebar`'s `getChats()` to re-execute on the next render. Called before `redirect()` so the new/updated sidebar data is available when the redirected page loads.

## 5. Copywriting

| Element | Copy | Notes |
|---------|------|-------|
| Sidebar "New Chat" button | `New Chat` | Preceded by `<Plus size={16} />` icon — icon is the visual verb |

**Tone:** Direct, minimal. No marketing language. No ellipses in button labels. Destructive actions use the verb of the action ("Delete"), not softened language ("Remove").
