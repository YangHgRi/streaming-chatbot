## Objective

Create the `src/app/actions.ts` Server Actions module with all three mutation functions (`createChatAction`, `renameChatAction`, `deleteChatAction`), then update the `Sidebar` Server Component to import those actions and pass them as props to `SidebarClient`. This is the final wiring step that makes all conversation CRUD operations functional end-to-end: the "New Chat" button creates and navigates, the inline rename persists the title to Postgres, and the delete confirm removes the conversation and all its messages (via CASCADE) before redirecting to root.

## must_haves

- `createChatAction` creates a new `Chat` row via `createChat()`, calls `revalidatePath('/', 'layout')`, then redirects to `/chat/${chat.id}` — sidebar immediately shows the new entry when the redirect lands.
- `renameChatAction(chatId, formData)` reads `formData.get('title')`, guards against blank/whitespace submissions, calls `updateChat(chatId, { title })`, and calls `revalidatePath('/', 'layout')` — updated title is visible in the sidebar without a manual page reload.
- `deleteChatAction(chatId)` calls `deleteChat(chatId)` (CASCADE removes all child message rows), calls `revalidatePath('/', 'layout')`, then redirects to `/` — the deleted conversation is removed from the sidebar and its messages are absent from the database.
- `redirect()` is NEVER called inside a `try/catch` block in any action — it throws a special Next.js internal error and must be called at the top level.
- All three actions are passed as props from the `Sidebar` Server Component to `SidebarClient` — Server Actions from `'use server'` modules are serializable and can be forwarded as props to Client Components.

<action>
Create a new file at `src/app/actions.ts`. This file is a Server Actions module: the `'use server'` directive at the top of the file marks every exported function as a Server Action automatically.

Write this exact content:

```ts
'use server';
import { redirect } from 'next/navigation';
import { revalidatePath } from 'next/cache';
import { createChat, updateChat, deleteChat } from '@/lib/db/queries';

// CONV-01: Create a new chat and navigate to it.
// revalidatePath is called BEFORE redirect so the sidebar data is fresh when the
// redirected page renders (Router Cache is busted before the new URL loads).
export async function createChatAction() {
  const chat = await createChat();
  revalidatePath('/', 'layout');
  redirect(`/chat/${chat.id}`);
}

// CONV-04: Rename a conversation.
// chatId is pre-bound via .bind(null, chat.id) in SidebarClient's form action.
// formData carries the 'title' input field value.
// Guard: returns early on blank/whitespace — prevents saving empty titles.
// No redirect — user stays on the current chat; sidebar title updates via revalidatePath.
export async function renameChatAction(chatId: string, formData: FormData) {
  const title = formData.get('title') as string;
  if (!title?.trim()) return;
  await updateChat(chatId, { title: title.trim() });
  revalidatePath('/', 'layout');
}

// CONV-05: Delete a conversation and all its messages.
// deleteChat uses CASCADE on the messages foreign key — all child message rows are
// removed automatically by Postgres. No separate deleteMessages() call needed.
// Redirects to '/' after deletion. The root page (app/page.tsx) will auto-create
// a new chat and redirect to it, so the user always lands in a conversation.
export async function deleteChatAction(chatId: string) {
  await deleteChat(chatId);
  revalidatePath('/', 'layout');
  redirect('/');
}
```

Critical constraints per RESEARCH §5.1 and RISK-02:
- `redirect()` is called at the top level in `createChatAction` and `deleteChatAction` — NOT inside `try/catch`. `redirect()` throws a special `NEXT_REDIRECT` error internally; if it were inside a `catch` block, the error would be swallowed and navigation would never happen.
- `revalidatePath('/', 'layout')` uses the `'layout'` type — this invalidates the Router Cache for the entire root layout subtree (not just one page). This causes the `Sidebar` Server Component's `getChats()` call to re-execute on next render, making the sidebar reflect the mutation immediately.
- `revalidatePath` is called BEFORE `redirect()` in create and delete actions — this ensures the cache is already busted when the browser navigates to the new URL.
- The `'use server'` directive is at the FILE level (first line, before imports) — this marks all exports as Server Actions without needing per-function directives.
</action>

Server Actions from `'use server'` modules are serializable function references and can be passed as props from Server Components to Client Components — this is the official Next.js pattern for giving Client Components access to Server Actions.

Replace the entire content of `src/components/Sidebar.tsx` with:

```tsx
// src/components/Sidebar.tsx
// Async Server Component — fetches all chats and passes Server Actions as props to SidebarClient.
// Re-fetches chats on every request after revalidatePath('/', 'layout') is called.
import { getChats } from '@/lib/db/queries';
import { SidebarClient } from './SidebarClient';
import {
  createChatAction,
  renameChatAction,
  deleteChatAction,
} from '@/app/actions';

export async function Sidebar() {
  const chats = await getChats();
  return (
    <SidebarClient
      chats={chats}
      createChatAction={createChatAction}
      renameChatAction={renameChatAction}
      deleteChatAction={deleteChatAction}
    />
  );
}
```

Key points:
- Imports come from `@/app/actions` (the `'use server'` module created in T01).
- All three actions are forwarded as props — `SidebarClient` already declares them in its `SidebarClientProps` interface (from plan 03-02).
- The `getChats()` call is unchanged from plan 03-02.
- No `'use client'` directive — this remains a Server Component.
</action>
